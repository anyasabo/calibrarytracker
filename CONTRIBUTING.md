# Contributing to CA Library Card Tracker

Thanks for helping! This guide covers everything you need to get started.

## Project Structure

```text
calibrarytracker/
├── cmd/scraper/          # Go CLI that fetches library data from CA State Library
│   └── main.go
├── internal/scraper/     # Go scraper internals (parsing, merging, output)
├── data/                 # Reference data (JSON) — output of the scraper, checked in
│   ├── systems.json      # All 186 CA library systems (scraper-generated)
│   ├── branches.json     # All ~1,130 branch locations (scraper-generated)
│   ├── cooperatives.json # CLSA cooperative system membership (hand-curated)
│   ├── partnerships.json # Reciprocal borrowing partnerships (hand-curated)
│   └── digital-access.json # OverDrive/Libby URLs and eCard info (hand-curated)
├── src/                  # SvelteKit frontend app
│   ├── lib/
│   │   ├── types/        # TypeScript type definitions (shared between scraper output + app)
│   │   ├── db/           # Dexie IndexedDB wrapper for user data
│   │   └── components/   # Reusable Svelte components
│   └── routes/           # SvelteKit file-based routing
├── static/               # Static assets (favicon, manifest, etc.)
├── .github/workflows/    # CI (lint, build, test) + scheduled scraper
└── tests/                # Playwright end-to-end tests
```

## Prerequisites

- **mise** (required) — use it to install and activate runtime versions from repo files.
  - Install mise using the official docs: <https://mise.jdx.dev/getting-started.html>
  - Enable mise shell activation in your shell profile (bash/zsh/fish), then open a new shell:
    <https://mise.jdx.dev/getting-started.html#activate-mise>
- If you cannot use mise, install these manually:
  - **Node.js 24.x** + npm
  - **Go 1.26**

## Quick Start

```bash
# 0) Confirm mise is active in your shell
mise --version

# 1) Install repo runtimes from version files
mise install

# 2) Confirm versions (should match .node-version and .go-version)
node -v
go version

# 3) Install frontend dependencies
npm ci --no-audit --no-fund

# 4) Refresh library data (scraper)
go run ./cmd/scraper

# 5) Run frontend checks
npm run lint
npm run check

# 6) Build production static site
npm run build
```

If you only touch scraper/data files, you do **not** need to run the frontend dev server.

## Development Workflow

### Frontend (SvelteKit)

The app is a static site (no server) that runs entirely in the browser.
User data lives in IndexedDB; reference data (library systems, branches)
is bundled as static JSON at build time.

```bash
npm run dev          # Start dev server on http://localhost:5173
npm run build        # Production build (static files in build/)
npm run build:pages  # Production build with GitHub Pages base path (/calibrarytracker)
npm run preview      # Preview the production build locally
npm run check        # TypeScript type checking via svelte-check
npm run lint         # ESLint + Prettier
npm run format       # Auto-fix formatting
npm run test         # Run browser smoke tests against a local preview server
npm run test:smoke   # Same as above (explicit script name)
```

### Scraper (Go)

The scraper downloads SpreadsheetML (XML) exports from countingopinions.com
(the CA State Library's official reporting platform), parses them, and
outputs JSON files into `data/`. It also merges in `digital-access.json`.

```bash
go run ./cmd/scraper              # Run the scraper
go test ./internal/scraper/...    # Run scraper tests
```

The scraper tries the Excel XML export first (faster, more stable), then
falls back to HTML scraping if the Excel download fails.

### Data Files

Files in `data/` are the source of truth for library reference data:

- **`systems.json`** and **`branches.json`** — generated by the scraper.
  Do not edit by hand; re-run the scraper instead.
- **`cooperatives.json`** and **`partnerships.json`** — hand-curated.
  Edit these directly when cooperative membership or partnerships change.
- **`digital-access.json`** — hand-curated. Maps system IDs to OverDrive
  URLs and eCard registration info. The scraper reads this file and merges
  the data into `systems.json` at build time.

## Key Architectural Decisions

### Why no backend?

All user data (cards, visits) is stored in the browser's IndexedDB.
This means:

- Free hosting on GitHub Pages (just static files)
- User data never leaves their device
- Works fully offline once loaded
- Users can export/import their data as JSON for backup

### Why SvelteKit with adapter-static?

SvelteKit's static adapter prerenders all pages at build time. Combined
with a service worker, the entire app works offline. Svelte's compiled
output is also very small (~5KB gzipped), which matters for a PWA.

### Why Go for the scraper?

The scraper is a standalone CLI tool that parses Excel/CSV/HTML. Go is
simple, compiles to a single binary, and has good libraries for all three
formats. It runs in CI as a GitHub Action.

## Code Standards

### TypeScript

- **Strict mode is on.** All types must be explicit; no `any` unless
  absolutely unavoidable (and then add a comment explaining why).
- Type definitions shared between the scraper output and the frontend
  live in `src/lib/types/`. If you change the JSON schema in the scraper,
  update the types to match.
- Use `interface` for object shapes, `type` for unions and intersections.

### Svelte Components

- One component per file. File name matches component name in PascalCase.
- Props use Svelte 5 `$props()` rune. Always type your props.
- Keep components small. If a component exceeds ~150 lines, extract
  sub-components.
- Use `$state` and `$derived` runes for reactivity. Avoid `$effect`
  when `$derived` will do — effects are harder to reason about.

### CSS

- Use Svelte's scoped `<style>` blocks. No global CSS except in
  `src/app.css` for resets and CSS custom properties (design tokens).
- Use CSS custom properties (`var(--color-primary)`) for theming.
  Never use raw color hex values in components.

### Go

- Standard `go fmt` formatting.
- Run `go vet` and `staticcheck` before committing.
- Errors are returned, not panicked. Log and exit gracefully.
- The scraper should be idempotent — running it twice produces identical
  output if the source data hasn't changed.
- Keep `.go-version` and `go.mod` aligned (CI verifies this).

## Linting & Formatting

CI runs all of these. Your PR will be blocked if any fail.

```bash
npm run lint         # ESLint (code quality) + Prettier (formatting)
npm run format       # Auto-fix Prettier issues
npm run check        # svelte-check (TypeScript + Svelte diagnostics)
```

**ESLint** catches common mistakes: unused variables, missing types,
accessibility issues in Svelte templates, etc.

**Prettier** enforces consistent formatting. Don't argue about tabs vs
spaces — just run `npm run format`.

**svelte-check** runs the TypeScript compiler and Svelte-specific
diagnostics. It catches type errors, invalid bindings, and accessibility
warnings in your templates.

## Data Model Reference

See the plan document or `src/lib/types/` for the full type definitions.
Quick summary:

- **LibrarySystem** — one of ~186 administrative entities (e.g., "Sacramento Public Library")
- **Branch** — a physical location within a system (~1,310 total)
- **CooperativeSystem** — a CLSA cooperative with subgroups (e.g., NorthNet → MVLS, North Bay, North State)
- **Partnership** — an explicit reciprocal borrowing agreement between systems
- **UserCard** — a library card the user holds (stored in IndexedDB, never sent anywhere)
- **BranchVisit** — a record that the user visited a specific branch

## Making Changes

1. Create a branch: `git checkout -b my-feature`
2. Make your changes
3. Run checks for the area you touched:
   - Frontend/app changes: `npm run lint && npm run check && npm run build && npm run test:smoke`
   - Scraper/Go changes: `go vet ./... && go test ./...`
4. Commit with a clear message explaining _why_ (not just _what_)
5. Open a PR — CI will run automatically

## Common Pitfalls

- **Don't import from `$app/server`** — this is a static site, there is
  no server. If you see yourself reaching for server-side code, you're
  going down the wrong path.
- **Don't store user data anywhere except IndexedDB** — no cookies, no
  localStorage (5MB limit), no server calls.
- **Don't add API keys** — maps use OpenStreetMap (free, no key needed).
  If a feature requires an API key, it needs a different approach.
- **Don't edit `data/systems.json` or `data/branches.json` by hand** —
  re-run the scraper. Hand edits will be overwritten.
- **Do edit `data/cooperatives.json` and `data/partnerships.json`** —
  these are hand-curated and not auto-generated.
